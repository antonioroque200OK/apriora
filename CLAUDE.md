# Apriora — CLAUDE.md

## Project Overview

Apriora is a Brazilian gamified learning platform. The repository is a pnpm + Turborepo monorepo containing:

- `apps/web` — Next.js 16 teacher interface + landing
- `apps/mobile` — React Native + Expo student experience
- `apps/api` — NestJS + Fastify backend
- `packages/shared` — Domain types, schemas, constants, helpers
- `packages/api-client` — Typed HTTP + Socket.IO client
- `packages/config` — Shared tsconfig and Biome config
- `packages/ui` — Placeholder (do not populate until Phase 3)

## Engineering Principles

### Inspect before modifying

Read and understand any file before changing it. Identify its purpose, its dependents, and the intent of the existing code. Never overwrite working code with assumptions.

### Specification-driven development

The development order is: **PRD → Architecture → Domain Model → Roadmap → Implementation.**
Do not implement product features before the relevant specification exists. The current phase is shown in [engineering-foundation.md](docs/foundation/engineering-foundation.md).

### Document decisions, not just outcomes

Before implementing any architectural or structural change, write an ADR in `docs/adr/`. The ADR must explain context, alternatives, trade-offs, and rationale. Update existing ADRs when decisions change.

### Preserve Kombai-generated UI

The UI in `apps/web/components/` may have been generated by Kombai. Preserve it unless you have an explicit, documented reason to change it. Prefer additive changes. Never delete or rewrite generated UI as part of an unrelated task.

### Avoid unrelated refactors

Fix the problem at hand. Do not refactor surrounding code unless it directly causes the issue. Incremental improvements over destructive rewrites.

### Keep documentation synchronized

When the architecture changes, update `docs/architecture/` and the relevant ADR in the same commit. When a new pattern is established, document it. Stale documentation is actively harmful.

## Code Quality

**Biome** is the single tool for formatting and linting (replaces ESLint + Prettier).

```bash
pnpm check       # format + lint + organize imports (writes fixes)
pnpm typecheck   # TypeScript across all packages
pnpm build       # Full build
pnpm ci:check    # CI mode — check without writing (used in CI)
```

**Before considering any task complete:**

1. `pnpm check` passes with no errors.
2. `pnpm typecheck` passes with no errors.
3. If the change touches a buildable app: `pnpm build` succeeds.
4. If tests exist for the affected area: `pnpm test` passes.

```bash

pnpm lint                              # Lint only (via Turbo)
```

## Architecture Quick Reference

| Tool | Role |
| --- | --- |
| pnpm workspaces | Package management and linking |
| Turborepo | Task orchestration and caching |
| Turbopack | Next.js dev bundler (default, no config needed) |
| Biome | Format + lint (single root `biome.json`) |

**Turborepo ≠ Turbopack.** Turborepo orchestrates tasks. Turbopack bundles `apps/web` during `next dev`. See [ADR-001](docs/adr/adr-001-monorepo-and-workflow.md).

**tRPC is explicitly ruled out.** See [ADR-002](docs/adr/adr-002-technology-stack.md) §Part 2. Do not propose or implement tRPC.

## Apps

### `apps/web`

- Next.js 16, App Router, TypeScript, Tailwind CSS 4, shadcn/ui (`base-nova`), `@base-ui/react`
- Dev port: **3000**
- Turbopack enabled by default — do not add custom Turbopack config unless there is a concrete, documented need
- `components.json` configures shadcn/ui path aliases — do not modify without understanding it

### `apps/mobile`

- React Native 0.79, Expo SDK 53, Expo Router 5
- Dev port: **8081** (Expo Metro)
- TypeScript only — no JavaScript files

### `apps/api`

- NestJS 11, Fastify adapter, Socket.IO, Prisma, Zod, JWT
- Dev port: **3001**
- Not yet implemented — stub only. Implementation begins in Phase 6.

## Packages

### `@apriora/shared`

The single source of truth for domain types, enums, constants, Zod schemas, and business-rule helpers. If a type is used by more than one app or package, it belongs here.

### `@apriora/api-client`

Typed HTTP + Socket.IO client. Both web and mobile import from here. Never call `fetch` directly from apps.

### `@apriora/config`

Exports:

- `@apriora/config/tsconfig/base`
- `@apriora/config/tsconfig/nextjs`
- `@apriora/config/tsconfig/react-native`
- `@apriora/config/biome` — extends root `biome.json` for per-package overrides

### `@apriora/ui`

Placeholder. Do not implement until Phase 3 (Software Architecture) identifies concrete cross-platform component sharing needs. See [ADR-002](docs/adr/adr-002-technology-stack.md) §1.2 and engineering foundation §6.

## Git and GitHub Workflow

Branch strategy:

```text
main       ← production (protected)
staging    ← integration
feature-*  ← work branches, created from staging
```

Flow: `feature-*` → PR → `staging` → PR → `main`

Use `gh` CLI for GitHub automation:

```bash
gh issue create --title "feat: ..." --label "feature"
gh issue develop <number> --base staging --checkout
gh pr create --base staging --fill
```
